# План нормализации базы кодов НЦТ

Дата анализа: 2026-07-15.

## Что уже изучено

Локальные источники:

- `pdf/11_class_all.pdf`: 69 страниц. По титульной странице это ЦВЭ 2025, четвертый период регистрации, оставшиеся места, только заочная и дистанционная формы на базе 11 классов. Этот файл нельзя считать полной базой приема 2026-2027.
- `pdf/9_class.pdf`: 37 страниц. Это ИМД 2026, первый период регистрации, перечень комбинаций специальностей для 2026-2027 на базе 9 классов.
- `src/data/new_db.json`: основной активный файл приложения, 1562 записи, структура `{ records, errors }`.
- `src/data/nct-codes.json`: старый/легаси файл, 125 записей, все по Худжанду, год 2021-2022. Его нельзя сливать с актуальной базой как равноправный источник без явного поля года.

Официальные онлайн-источники НЦТ:

- `https://ntc.tj/tj/ba-dovtalab/nakshai-kabul.html` - официальная страница "Нақшаи қабул 2026-2027"; содержит ссылки на статистическую таблицу, систему выбора специальностей и PDF по группам.
- `https://stat.ntc.tj/Y26/RPlan` - официальная HTML-таблица плана приема 2026-2027. В ней есть поля: ID, группа ихтисосов, муассисаи таълимӣ, рамзи ихтисос, номи ихтисос, шакли таҳсил, намуди таҳсил, маблағи таҳсил, забони таҳсил, нақша, зинаи таҳсил.
- `https://spec.ntc.tj/` - официальный интерфейс выбора специальностей 2026-2027.
- `https://ntc.tj/tj/elon1-3.html` - официальная страница плана приема 2026-2027 для второго периода регистрации.
- `https://ntc.tj/tj/ba-dovtalab/rohnamoi-dovtalab.html` - "Роҳнамои довталаб - 2026"; нужен для правил экзаменов, периодов, шкалирования, выбора специальности, льгот и логики конкурса распределения.
- `https://ntc.tj/tj/ba-dovtalab/muassisaho.html` - официальный перечень учреждений.

## Текущие проблемы в базе

Проверка `src/data/new_db.json` показала:

- Всего записей: 1562.
- Точных дублей по текущим основным полям не найдено.
- 487 записей имеют пустую `location`.
- Есть загрязненные локации: например `город Турсунзаде заочная`, `город Пенджикент заочная`, `район Кушониён дистанционная платный (3295) таджикский`.
- Есть строки, где часть специальности попала в `university_name`, например `хозяйственной техники ...`.
- Есть строки, где город добавлен в конец названия вуза, например `... шаҳри Хуҷанд Хуҷанд`.
- Часть данных после 9 класса на таджикском, часть после 11 класса на русском, а 11-классный PDF относится к 2025/четвертому периоду, поэтому сейчас база смешивает языки, годы и периоды.
- Типы приложения пока ожидают обязательные строки для `location`, `education_form`, `education_type`, `tuition_fee`, `language`, хотя в реальных данных есть `null`/потерянные значения.
- В `src/lib/db/types.ts` часть констант отображается mojibake-текстом, это отдельная кодировочная проблема интерфейса/типов.

## Почему Худжанд выглядит неполным

Проблема не только в количестве записей. В текущей базе есть 97 записей с `location = "город Худжанд"`, но часть худжандских записей не попадает в этот счетчик, потому что город находится внутри `university_name`, например:

- `Худжандский международный институт ... город Худжанд`
- `Коллеҷи тиббии шаҳри Хуҷанд ба номи Ю. Б. Исҳоқӣ Хуҷанд`
- `Коллеҷи сайёҳӣ ва хизматрасониҳои иҷтимоии шаҳри Хуҷанд Хуҷанд`

Значит фильтр по городу недосчитывает записи, а исправление должно включать восстановление города из официального поля `макон`, а не простую замену строк.

## Целевая схема записи

Минимальный формат для исправленной базы:

```ts
type NctAdmissionRecord = {
  id: string
  academic_year: string
  admission_period: "first" | "second" | "third" | "fourth" | "unknown"
  source_kind: "official_stat" | "official_pdf" | "official_spec" | "legacy"
  source_url?: string
  source_document?: string
  source_page?: number
  source_row_id?: string

  nct_id?: string
  code: string
  specialty_name: string
  specialty_name_original: string
  specialty_name_normalized: string
  university_name: string
  university_name_original: string
  university_name_normalized: string
  city: string | null
  city_normalized: string | null
  region?: string | null

  education_level: "after_9" | "after_11"
  cluster: 1 | 2 | 3 | 4 | 5
  cluster_name: string
  education_form: "очная" | "заочная" | "дистанционная" | null
  education_type: "бесплатный" | "платный" | null
  tuition_fee: number | null
  language: string[]
  admission_plan: number | null

  direction: string | null
  profession: string[]
  taxonomy_node_ids: string[]
  keywords: string[]
  synonyms: string[]
  search_text: string

  confidence: number
  confidence_reasons: string[]
  review_status: "verified" | "needs_review" | "conflict" | "incomplete"
}
```

Стабильный `id` должен строиться из фактического варианта приема, а не только из кода специальности:

```text
nct:<academic_year>:<period>:<level>:<cluster>:<source_row_id-or-hash>
```

Хеш допустим только от нормализованного набора:

```text
code + university_name_normalized + city_normalized + education_form + education_type + language + admission_plan + source
```

Так разные коды одной специальности и разные формы/типы обучения не будут ошибочно объединены.

## Правила нормализации

1. Канонический источник для 2026-2027: `stat.ntc.tj/Y26/RPlan`, затем PDF НЦТ, затем `spec.ntc.tj`.
2. Локальный `11_class_all.pdf` использовать как отдельный источник 2025/четвертый период, не как замену полному плану 2026-2027.
3. Точные дубли удалять только при полном совпадении ключевых полей и источника. Разные формы, языки, планы, типы обучения или разные `source_row_id` сохранять отдельно.
4. Города восстанавливать из официального поля `макон` или из перечня учреждений. Если город извлечен только из хвоста `university_name`, ставить `confidence <= 0.7` и `needs_review`.
5. Названия вузов и городов нормализовать через справочники aliases -> canonical, сохраняя исходную строку.
6. Специальности нормализовать осторожно: убрать переносы и явные склейки строк, но не переводить и не дописывать смысл без источника.
7. Язык хранить массивом (`["таджикский"]`, `["таджикский", "русский"]`), потому что в НЦТ бывают смешанные варианты.
8. Стоимость хранить числом в сомони или `null`, отдельно от `education_type`.
9. Все строки с потерянной колонкой, пустым планом, неизвестным городом, конфликтом года/периода или расхождением PDF/stat отправлять в отчет `needs_review`.

## Уровни уверенности

- `0.95-1.00`: запись совпала между official_stat и PDF или взята напрямую из official_stat без аномалий.
- `0.85-0.94`: запись взята из одного официального источника, все обязательные поля заполнены и проходят проверки типов.
- `0.65-0.84`: запись восстановлена из PDF, но требовала склейки переносов/колонок или город был выведен из названия учреждения.
- `<0.65`: неполная или конфликтная запись, не должна участвовать в рекомендациях без ручной проверки.

## План работ после утверждения схемы

1. Добавить скрипт инвентаризации источников: PDF metadata, страницы, строки, контрольные суммы, год/период.
2. Добавить загрузчик official_stat с кэшированием сырого HTML/JSON/CSV в `data/raw/ntc/`.
3. Перепарсить PDF таблицы по координатным колонкам и сверить с HTML-таблицей НЦТ.
4. Сформировать `data/intermediate/nct_candidates.json` с сырыми кандидатами и provenance.
5. Прогнать нормализацию городов, вузов, форм, типов, языков, стоимости, плана, уровня образования.
6. Сгенерировать:
   - исправленную базу;
   - отчет проблем;
   - отчет по Худжанду;
   - diff старой и новой базы;
   - методологию проверки.
7. Только после просмотра отчетов заменить `src/data/new_db.json` или подключить новую версионированную базу рядом.

## Проверки качества

Обязательные проверки перед заменой базы:

- количество записей по источникам, уровню, кластеру, городу, вузу;
- список записей с пустым городом, формой, типом, языком, планом;
- сравнение counts по `stat.ntc.tj/Y26/RPlan` и локальной базе;
- проверка, что разные коды одной специальности не объединены;
- проверка, что один и тот же код в разных вузах/формах сохранен;
- отдельный счетчик по Худжанду: direct city, city recovered from institution, unresolved;
- sample audit 10-20 строк на каждом проблемном типе через визуальную страницу PDF.

## Функции продукта, которые стоит добавить на базе PDF/НЦТ

1. Карта поступления по НЦТ: кластер -> предметы -> код -> вуз -> город -> форма -> бюджет/платно -> план приема.
2. Фильтр "реально доступно мне": после 9/11, город, язык, форма обучения, бюджет/платно, кластер.
3. Объяснение рекомендации через официальный источник: "почему этот код", "какой кластер", "какие экзамены", "какой план приема", "откуда данные".
4. Режим "Худро бисанҷ": мини-диагностика по предметам кластера, связанная с рекомендациями.
5. Таймлайн ИМД: регистрация, приглашение, экзамены, результаты, апелляция, конкурс распределения.
6. Симулятор выбора комбинаций: пользователь собирает список кодов и видит риски по городу, конкуренции, форме, стоимости и запасным вариантам.
7. Отдельный блок творческих/специальных экзаменов, медицинских ограничений и льгот, потому что эти правила есть в официальных разделах НЦТ и влияют на выбор.
8. Страница источника записи: открыть PDF/страницу/строку, показать confidence и причину спорности.
9. Админ-аудит данных: список `needs_review`, быстрые фильтры по типу ошибки, подтверждение/отклонение исправления.
10. Версионирование базы по учебному году и периоду регистрации, чтобы 2025/четвертый период не смешивался с 2026-2027/первым периодом.
