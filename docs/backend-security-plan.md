# Backend Security and Admin Control Plan

## Goal

Build a secure backend architecture for `nct-navigator-back` that:

- does not trust any arbitrary client that knows the backend URL
- protects expensive and sensitive routes from abuse
- allows only approved frontend flows to call internal backend capabilities
- supports origin checks, HMAC request signing, authentication, and rate limiting
- provides an admin dashboard for runtime control
- allows temporary server shutdown with a graceful frontend maintenance screen
- preserves current frontend behavior during rollout and avoids breaking existing flows

## Current Situation

The backend is publicly reachable on Vercel and currently behaves like a normal public HTTP service. That means:

- if someone knows the backend URL, they can attempt to call it directly
- origin-based protection alone is not sufficient
- some routes already require Supabase session auth, but that only protects authenticated user flows
- some routes are public or partially public and can still be abused
- there is no central request policy layer
- there is no global maintenance mode
- there is no centralized rate limit control
- there is no admin dashboard for operational actions

## Main Security Principle

We will not rely on only one security mechanism.

The final protection model will use multiple layers:

1. Route classification
2. Origin and Referer validation for browser-originated requests
3. HMAC signatures for internal trusted requests
4. Supabase user authentication for user-specific routes
5. Admin role authorization for operational controls
6. Rate limiting by route group, IP, user, and signature identity
7. Maintenance mode and kill switches
8. Audit logging and admin observability

This layered design is important because:

- `Origin` headers help for browsers, but do not stop server-to-server abuse
- a secret header alone is powerful, but should not be the only protection
- auth alone protects users, but not public AI abuse
- rate limiting reduces abuse but does not define trust

## Non-Goals

This phase will not try to:

- hide the backend URL completely
- make the backend private at the infrastructure level only
- replace all app-level controls with Vercel-only controls
- rewrite the entire frontend/backend contract at once

## High-Level Architecture

### Request Trust Levels

Every backend route will belong to one of these classes:

### 1. Public

Examples:

- health checks
- selected read-only public content

Rules:

- optional origin check
- strict rate limiting
- no sensitive actions

### 2. Authenticated User

Examples:

- user profile sync
- user-specific plans
- chat sessions
- coach history

Rules:

- valid Supabase session required
- optional origin validation
- per-user and per-IP rate limiting

### 3. Internal Trusted Frontend

Examples:

- expensive AI generation endpoints
- recommendation generation
- plan generation
- endpoints that should only be callable through your own frontend server layer

Rules:

- must come through frontend-controlled server code
- HMAC signature required
- timestamp required
- nonce or replay protection required
- rate limiting per signature key and per IP

### 4. Admin

Examples:

- maintenance mode toggle
- route access policy changes
- rate limit policy changes
- server disable/enable actions
- audit log viewing

Rules:

- authenticated admin user required
- admin role required
- sensitive audit log entries recorded
- stronger rate limits

## Why HMAC Is Necessary

Your exact concern is correct: if another app or service copies your backend URL, the backend currently cannot distinguish that service from your own frontend unless a stronger trust mechanism is added.

HMAC solves this by proving:

- the request was generated by a trusted party that knows the signing secret
- the request body and metadata were not changed after signing
- the request was created recently

### HMAC Model

The frontend server layer will create these headers:

- `x-nct-key-id`
- `x-nct-timestamp`
- `x-nct-nonce`
- `x-nct-signature`
- optional `x-nct-client-version`

The signature will be computed over a canonical string such as:

```text
METHOD
PATH
TIMESTAMP
NONCE
SHA256(BODY)
```

The backend will:

- reconstruct the canonical payload
- compute HMAC using shared secret
- compare in constant time
- reject if timestamp is too old
- reject if nonce was already used

### Important Limitation

Do not put the HMAC secret into browser-exposed frontend code.

That means:

- browser client components must not sign requests directly
- the frontend server layer must sign requests
- sensitive backend endpoints must be called through frontend route handlers or server actions

## Required Frontend Architecture Change

To safely use HMAC, we should not depend only on `next.config.ts` rewrites for sensitive routes.

### Why

Current rewrites are convenient, but they cannot safely implement rich signing logic for each request body and timestamp.

### New Model

For sensitive route groups:

- browser calls frontend route handler
- frontend route handler validates request shape
- frontend route handler signs outbound request to backend
- backend verifies HMAC

For lower-risk routes:

- rewrites may temporarily remain during migration

## Safe Rollout Strategy

We must avoid breaking the frontend after an update. To do that, rollout will happen in compatibility mode.

### Compatibility Rule

Sensitive backend routes will temporarily support two modes:

1. legacy trusted mode for existing frontend paths
2. signed mode for the new frontend proxy layer

Then, after frontend deployment is stable:

- we disable legacy mode per route group

### Migration Sequence

1. Add backend security framework in passive mode
2. Add frontend signed proxy handlers
3. Switch selected frontend calls to signed handlers
4. Observe logs
5. Enforce HMAC on selected routes
6. Remove legacy route access for those routes

## Route Grouping Plan

The backend should be grouped into policy families.

### Group A: Public low-risk

Possible examples:

- simple health endpoint
- maintenance status endpoint

Policy:

- no auth
- no HMAC
- strict rate limit

### Group B: Public expensive AI and compute

Current likely candidates:

- recommendations
- analyze
- generate-plan
- coach chat if exposed without session

Policy:

- move behind signed frontend proxy
- optional auth depending on flow
- aggressive rate limit

### Group C: User-authenticated

Examples:

- plan fetch/update
- chat sessions
- sync profile
- coach persistence

Policy:

- Supabase session required
- route-level rate limit
- optional origin validation

### Group D: Admin operations

Examples:

- toggle maintenance
- update allowed origins
- update rate limits
- update route policies
- emergency disable route groups

Policy:

- admin auth required
- audit logging required
- optional step-up confirmation for dangerous actions

## Server Shutdown Behavior

This is one of the most important product requirements.

When you press `Close Server` in admin:

1. backend state changes to maintenance mode
2. backend starts returning a controlled response for frontend traffic
3. frontend recognizes that response
4. frontend shows a dedicated maintenance page or full-screen message
5. when server is re-enabled, frontend resumes normal behavior

### Required Backend Response Contract

The backend should return a consistent machine-readable maintenance response.

Example:

```json
{
  "status": "maintenance",
  "code": "SERVER_DISABLED",
  "message": "Site is temporarily unavailable",
  "retryAfter": 300
}
```

Recommended HTTP status:

- `503 Service Unavailable`

Recommended headers:

- `Retry-After`
- `Cache-Control: no-store`

### Required Frontend Behavior

The frontend must:

- detect maintenance responses from backend
- stop showing generic crash states
- render a designed maintenance screen
- optionally preserve current page URL
- retry on manual refresh or after server re-enable

### Frontend Resilience Requirement

The maintenance screen must be implemented in a way that:

- existing components do not all need custom handling
- one shared fetch wrapper or proxy layer can trigger maintenance UI
- hydration and route navigation remain stable

## Recommended Frontend Maintenance Design

Implement a centralized maintenance handling flow:

- create a shared frontend API client wrapper
- when response status is `503` and code is `SERVER_DISABLED`, update a global maintenance store
- render a top-level maintenance boundary in the frontend app shell

Behavior:

- normal app hidden
- branded maintenance screen shown
- message text customizable from backend/admin dashboard later

## Rate Limiting Plan

Rate limiting should be configurable and centralized.

### Dimensions

Apply limits by:

- route group
- IP address
- authenticated user id
- internal key id

### Storage Options

Best option:

- Redis or Upstash Redis for counters and nonces

Fallback option:

- Supabase table, but this is weaker for high-frequency rate limiting

### Policies

Examples:

- `recommendations`: low burst, moderate sustained
- `generate-plan`: strict
- `coach/chat`: strict and cost-aware
- `auth/login`: anti-bruteforce
- `admin/*`: low frequency but safe

### Admin Configurability

Dashboard should allow:

- enable/disable rate limit policy
- change per-minute quota
- change burst window
- temporary emergency tightening

## Origin Allowlist Plan

Origin allowlists are still useful, but must be treated as one layer only.

### Use Cases

- block random browser-originated cross-site calls
- restrict preview deployments and production domains
- reduce casual abuse

### Admin Control

Dashboard should allow:

- add origin
- disable origin
- label origin type
- production, preview, local dev, temporary

### Important Note

Origin checks should not be the only trust check for internal or sensitive routes.

## Replay Protection Plan

HMAC without anti-replay is incomplete.

We will add:

- short timestamp validity window, for example 60 to 120 seconds
- nonce storage in Redis
- reject repeated nonce for the same key within validity window

This prevents an intercepted signed request from being replayed repeatedly.

## Admin Authorization Plan

Admin dashboard access must not be based on email string checks in the frontend only.

Recommended model:

- Supabase auth login
- backend verifies authenticated user
- backend checks admin role from database or claims

### Suggested Data Model

Table or policy source:

- `admin_users`
- or `user_roles`

Fields:

- `user_id`
- `role`
- `is_active`
- `created_at`
- `updated_at`

## Configuration Storage Plan

Runtime-managed settings should not live only in `.env`.

### Split Configuration Into Two Types

#### 1. Static Secrets

Remain in env:

- HMAC secrets
- Supabase service role
- private API keys

#### 2. Runtime Operational Settings

Stored in database:

- maintenance mode
- maintenance message
- allowed origins
- route policy flags
- rate limit policies
- admin dashboard preferences

This allows the dashboard to change behavior without redeploying.

## Proposed Database Tables

Suggested new tables:

### `security_settings`

Stores singleton-like global flags:

- backend enabled
- maintenance enabled
- maintenance message
- maintenance retry seconds

### `allowed_origins`

Stores origin allowlist entries:

- id
- origin
- environment
- is_active
- notes

### `route_security_policies`

Stores per route group policy:

- route_group
- auth_mode
- require_hmac
- require_origin
- enabled
- rate_limit_policy_id

### `rate_limit_policies`

Stores limits:

- policy name
- window seconds
- max requests
- burst
- enabled

### `security_audit_logs`

Stores decisions and admin actions:

- event_type
- route
- ip
- user_id
- origin
- key_id
- decision
- reason
- metadata
- created_at

### `hmac_keys`

Stores active key ids and metadata:

- key_id
- label
- status
- rotation metadata

Do not store plaintext secrets in a user-manageable dashboard table if avoidable. Key rotation metadata can be stored, but actual secret material should remain in secure env or secret manager.

## Key Management Plan

HMAC keys need a rotation strategy from day one.

### Required Features

- support `current` and `next` key
- accept multiple active key ids during rotation
- identify key by `x-nct-key-id`
- allow zero-downtime key rotation

### Rotation Flow

1. add new key to env and backend trust list
2. switch frontend signing to new key id
3. observe successful traffic
4. disable old key

## Frontend Compatibility Strategy

We must avoid a frontend crash after backend hardening.

### Rules

- never switch all routes to strict HMAC in one deployment
- start with admin and high-cost routes
- keep legacy behavior temporarily for routes still used directly
- add frontend-level maintenance handling before turning on server shutdown control
- introduce one centralized request abstraction before mass migration

### Frontend Tasks Needed

- add signed proxy route handlers for sensitive backend endpoints
- add global maintenance state
- add maintenance page or shell overlay
- add shared API error handling
- keep current UI contract stable where possible

## Risks and How We Avoid Them

### Risk 1: Frontend breaks after backend hardening

Mitigation:

- compatibility mode
- route-by-route rollout
- signed proxy introduction before enforcement

### Risk 2: Browser code leaks secrets

Mitigation:

- HMAC signing only in server-side frontend code
- no secrets in client bundles

### Risk 3: Maintenance mode causes random broken states

Mitigation:

- one consistent backend error contract
- one centralized frontend maintenance handler

### Risk 4: Rate limiting blocks normal users

Mitigation:

- group-specific limits
- staged rollout
- admin overrides

### Risk 5: Overuse of database for hot path controls

Mitigation:

- cache runtime policies
- use Redis for counters and nonce tracking

## Final Expected Result

After implementation:

- arbitrary third-party services will not be able to freely use sensitive backend routes
- internal backend routes will accept only signed trusted traffic
- authenticated user routes will remain user-bound
- admin will be able to disable the backend or selected route groups without redeploy
- frontend will show a polished maintenance screen instead of crashing
- rate limits and access policies will be manageable from an admin dashboard
- rollout will be staged so the frontend does not suddenly fail after deployment
