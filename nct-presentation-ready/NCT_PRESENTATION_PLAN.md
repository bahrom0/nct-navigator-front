# NCT Navigator — план реализации веб-презентации

## 1. Цель

Создать отдельную интерактивную веб-презентацию для защиты проекта **NCT Navigator** на олимпиаде.

Презентация должна:

- работать прямо внутри текущего Next.js-проекта;
- открываться по отдельному маршруту `/presentation`;
- выглядеть как дорогая продуктовая презентация, а не обычный лендинг;
- укладываться в выступление продолжительностью **3–5 минут**;
- использовать реальные факты из проекта и репозитория;
- быть удобной для показа с клавиатуры без мыши;
- автоматически попасть в Vercel после коммита и push в ветку `main`.

## 2. Основные решения

### Формат

Не PowerPoint и не длинная маркетинговая страница.

Это полноэкранная веб-презентация из шести последовательных экранов:

1. Вступление
2. Проблема
3. Решение
4. Как работает алгоритм
5. Технологии и архитектура
6. Итог и переход к проекту

Каждый экран занимает почти весь viewport и работает как отдельный слайд.

### Маршрут

```text
/presentation
```

### Основная ссылка проекта

```text
https://mmtai.xyz
```

### Название

```text
NCT Navigator
```

Во всех текстах, заголовках и интерфейсе использовать только это название.


### Слайд 2 — Проблема

Основная мысль:

```text
Тысячи вариантов.
Ни одного уверенного ответа.
```

Карточки:

- **Информации много:** статьи, советы, списки вузов и сотни кодов, но почти нигде нет персонального ответа.
- **Маршрута нет:** абитуриент готовится ко всему сразу, тратит время и не понимает, на чём сфокусироваться.
- **Главный вопрос:** «Как понять, что этот выбор действительно мой?»

Основной контекст раскрывается голосом, а не длинными абзацами на экране.

### Слайд 3 — Решение

Основная мысль:

```text
От интересов пользователя
до конкретного кода НЦТ.
```

Центральный элемент: большой реальный скриншот NCT Navigator.

Файл изображения:

```text
public/photos/product-main.png
```

Три карточки:

- **Подбирает код:** учитывает город, уровень образования, интересы и выбранные направления.
- **Объясняет выбор:** показывает, какие интересы совпали и почему вариант оказался в выдаче.
- **Строит план:** превращает выбранное направление в конкретные следующие шаги подготовки.

### Слайд 4 — Как работает система

Основная мысль:

```text
ИИ помогает понять намерение.
База сохраняет правду.
```

Pipeline:

```text
Профиль пользователя
        ↓
AI-анализ интересов
        ↓
Жёсткие фильтры
        ↓
Локальный поиск по базе НЦТ
        ↓
Локальное ранжирование
        ↓
AI rerank и объяснение
        ↓
Персональный план
```

Фактическая логика из репозитория:

1. AI анализирует интересы пользователя.
2. AI не выбирает коды НЦТ на первом этапе.
3. Система формирует профессии, направления и поисковые намерения.
4. Жёсткие ограничения должны применяться до ранжирования.
5. Город и уровень образования не должны silently ослабляться.
6. Локальный движок ищет только существующие записи.
7. AI получает только короткий список валидных кандидатов.
8. AI может менять порядок и объяснять результат.
9. AI не может придумать новый код, изменить город, выйти за shortlist или нарушить hard constraints.

Ключевая формулировка:

```text
AI выбирает профессиональное намерение.
Локальный движок выбирает валидные варианты НЦТ.
AI только переранжирует и объясняет короткий список.
```

### Слайд 5 — Технологии и архитектура

Основная мысль:

```text
Интерфейс, данные и интеллект работают вместе.
```

Технологии:

- Next.js 16 — страницы, API и серверная логика;
- React 19 — интерактивные элементы интерфейса;
- TypeScript — контроль типов и снижение количества ошибок;
- Tailwind CSS 4 — визуальная система и адаптивность;
- Zustand — клиентское состояние;
- Supabase — профили, планы, история и данные пользователя;
- DeepSeek — анализ интересов, финальный rerank и объяснение;
- Framer Motion — плавные переходы и анимации.

Актуальная архитектурная цепочка:

```text
analyze-interest-profile.ts
            ↓
nct-match.ts
            ↓
rank-nct.ts
            ↓
finalize-recommendations.ts
            ↓
generate-plan.ts
```

Также можно упомянуть orchestration-файл:

```text
src/lib/recommendations/service.ts
```

Не показывать слишком много кода и не объяснять каждую библиотеку как в учебнике.

### Слайд 6 — Итог

Основная мысль:

## 6. Анимации

Использовать анимацию только там, где она помогает воспринимать материал.

Разрешённые эффекты:

- fade;
- лёгкий translateY;
- небольшой blur reveal;
- subtle scale;
- последовательное появление pipeline;
- плавная прокрутка;
- аккуратные hover-эффекты.

Не использовать:

- бесконечно летающие элементы;
- хаотичные частицы;
- кислотные градиенты;
- чрезмерный glassmorphism;
- бессмысленные 3D-эффекты;
- тяжёлые анимации, которые могут лагать на проекторе.

Обязательно учитывать `prefers-reduced-motion`.

## 10. Адаптивность

Основной режим:

```text
16:9
```

Проверить:

- 1366×768;
- 1920×1080;
- планшет;
- телефон.

На маленьком экране:

- карточки переходят в одну колонку;
- навигация упрощается;
- текст не обрезается;
- горизонтальный scroll отсутствует;
- презентация остаётся читаемой.

## 11. Техническая интеграция

### AppShell

Маршрут `/presentation` должен отображаться без обычного header приложения, профиля, чат-навигации и других элементов продукта.

Добавить отдельную проверку:

```ts
const isPresentationRoute = pathname === "/presentation";
```

И вернуть презентацию без стандартного shell.

### Не ломать текущий проект

Запрещено:

- менять существующую главную страницу без необходимости;
- переписывать глобальные стили продукта;
- удалять пользовательские изменения;
- менять алгоритм рекомендаций;
- трогать базу НЦТ;
- менять Supabase-схему;
- менять существующие API.

Все изменения должны быть изолированы внутри презентации.

## 12. Контент и анти-AI-slop правила

Не использовать:

- «инновационное решение нового поколения»;
- «революционный подход»;
- «раскрывает потенциал»;
- «будущее уже сегодня»;
- «уникальная экосистема»;
- «персонализированный опыт» без конкретного объяснения;
- длинные определения библиотек;
- повторение одной мысли разными словами.

Использовать:

- конкретные факты;
- короткие глаголы;
- понятные причинно-следственные связи;
- реальные ограничения;
- человеческие вопросы;
- формулировки со стороны абитуриента.

Хорошо:

```text
Пользователь выбирает город и уровень образования.
Система сначала отсекает неподходящие записи.
Только после этого начинается ранжирование.
```

Плохо:

```text
Наша инновационная AI-платформа осуществляет глубокий персонализированный анализ образовательной траектории пользователя.
```

## 14. Критерии готовности

Задача считается завершённой, когда:

- `/presentation` открывается в production;
- есть 6 полноэкранных слайдов;
- презентация управляется клавиатурой;
- fullscreen работает;
- progress bar работает;
- стандартный AppShell не мешает;
- логотипы выглядят корректно;
- настоящий QR ведёт на mmtai.xyz;
- placeholders можно заменить файлами без переписывания кода;
- все факты соответствуют репозиторию;
- build проходит;
- commit находится в `main`;
- Vercel успешно задеплоил изменения;
- презентацию можно рассказать за 3–5 минут;
- визуально это продуктовая презентация, а не AI-generated landing page.
